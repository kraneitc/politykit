# Charterfall v0.0.1: Game Design Brief Implementation

## Source

This implementation slice covers item `1. Game Design Brief` from [Milestone 0: Product Framing](milestone-0-product-framing.md).

## Goal

Create a concise design brief for the long-term Constitutional Survival City Builder so Charterfall has a clear north-star product context before prototype work begins.

The brief should be short enough to read in under five minutes and concrete enough to keep Milestone 1 from drifting into a full city builder implementation.

## Target Artifact

Create:

```text
docs/charterfall/design/constitutional-survival-city-builder.md
```

This artifact is product/design documentation, not code. It should describe the long-term game concept and clearly separate the north-star city builder from the first Charterfall vertical slice.

## Implementation Scope

The design brief must include the following sections.

### 1. Product Description

Add a one-paragraph description of the Constitutional Survival City Builder.

The paragraph should explain that the player manages a fragile settlement where institutional rules shape how scarce resources are allocated, how crises are handled, and how the community adapts after failure.

### 2. Core Fantasy

Describe the player as a constitutional architect for a fragile settlement.

The player should feel responsible for designing civic machinery, not only placing buildings or directly assigning every resource.

Required beats:

- The player builds and stewards a settlement.
- The player's deeper power is drafting and revising institutions.
- Crises reveal whether those institutions protect, exclude, delay, overload, or adapt.

### 3. Core Loop

Include this loop exactly once as the shared gameplay frame:

```text
Build institutions -> face crises -> inspect consequences -> revise the system -> try again
```

Then explain each step in one or two sentences:

- **Build institutions**: select or amend civic rules.
- **Face crises**: run a scarcity, shock, or legitimacy challenge.
- **Inspect consequences**: review metrics, events, harms, bottlenecks, and citizen reactions.
- **Revise the system**: amend rules based on the inquiry.
- **Try again**: continue, rerun, or compare a different institutional configuration.

### 4. Five Pillars

Document these five pillars with short explanations:

1. **Visible settlement**
   The long-term game should make citizens, districts, resources, infrastructure, and ecological constraints visible.

2. **Playable institutions**
   The player should draft and amend rules for allocation, authority, transparency, accountability, appeals, local autonomy, and emergency power.

3. **Crisis-driven simulation**
   Seasons, disasters, shortages, corruption events, migration, housing loss, public health needs, and supply breakdowns should test the system.

4. **Inspectable consequences**
   Outcomes should be shown through metrics, event timelines, citizen stories, district changes, public inquiries, and comparison reports.

5. **Revision over optimization**
   Failure should be informative. The game should reward learning, amendment, reruns, stress tests, and comparison rather than a single universal best answer.

### 5. Scope Distinction

Add a section that distinguishes the long-term city builder from Charterfall.

Use this framing:

- **Constitutional Survival City Builder** is the north-star game: spatial settlement management plus institutional design.
- **Charterfall** is the first vertical slice: a compact roguelike settlement governance prototype that proves the institutional loop before full city-building systems are added.
- **PolityKit** is the reusable simulation framework underneath both.

### 6. Responsibility Boundaries

Include a table that separates PolityKit responsibilities from game-layer responsibilities.

Required entries:

| Area | PolityKit responsibility | Game-layer responsibility |
|---|---|---|
| Simulation | Deterministic runs, seeds, models, metrics, events | Present outcomes as playable feedback |
| Institutions | Model and preset behavior, parameters, manifests | Clause UI, unlocks, wording, player choices |
| Scenarios | Scenario inputs, shocks, validation | Campaign framing, crisis order, tutorial context |
| Results | Dashboard, metrics, events, comparisons | Public inquiry screen, narrative reactions, outcome badges |
| AI | Optional advisory artifacts only | Do not treat AI text as authoritative simulation data |

### 7. Non-Goals For The First Slice

State that Milestone 1 is not expected to build:

- Full spatial construction.
- Full save/load.
- Deep population simulation.
- Procedural narrative campaigns.
- Full district simulation.
- Real-time city management.
- Multiplayer.
- Historical or real-world scenario simulation.
- A universal score proving one institution is best.

### 8. Claims Boundary

Include this exact wording:

```text
Charterfall shows how fictional institutional rules behave inside declared simulation assumptions. It does not prove that a real-world political, economic, or social system is superior.
```

Also state that the long-term city builder should use fictional settlements, fictional factions, inspectable assumptions, and comparison language such as "under these assumptions."

## Out Of Scope

This slice should not create:

- A playable app.
- UI mockups.
- Clause catalogs.
- Scenario files.
- API request examples.
- Win/fail thresholds.
- Machine-readable content.

Those belong to later Milestone 0 implementation slices.

## Acceptance Criteria

v0.0.1 is complete when:

1. `docs/charterfall/design/constitutional-survival-city-builder.md` exists.
2. The brief can be read in under five minutes.
3. The brief includes a one-paragraph product description.
4. The brief includes the core fantasy of the player as a constitutional architect for a fragile settlement.
5. The brief includes the shared core loop.
6. The brief documents all five pillars.
7. The brief distinguishes the long-term city builder from Charterfall.
8. The brief distinguishes PolityKit responsibilities from game-layer responsibilities.
9. The brief names first-slice non-goals.
10. The brief includes the required claims boundary wording.
11. The brief does not promise features that Milestone 1 is not expected to build.

## Verification

Manual verification is sufficient for this documentation slice:

1. Read the target artifact from top to bottom.
2. Confirm every acceptance criterion is represented.
3. Confirm the brief does not require raw knowledge of PolityKit internals.
4. Confirm the language is assumption-bound and avoids real-world proof claims.

No build or test run is required unless code changes are added in the same work item.
