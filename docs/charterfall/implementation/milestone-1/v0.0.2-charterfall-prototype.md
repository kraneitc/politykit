# Charterfall v0.0.2: Prototype Implementation

## Source

This implementation slice covers `Milestone 1: Charterfall Prototype` from [Charterfall Roadmap](../charterfall-roadmap.md).

## Goal

Create the first playable Charterfall prototype: a lightweight text-and-dashboard experience where a player can draft a charter, run a crisis through PolityKit, inspect consequences, amend the charter, compare the amended outcome, and continue through a three-crisis campaign.

The prototype should answer one question:

```text
Is it fun to draft institutions, watch them fail or adapt under crisis, and revise them based on inspectable consequences?
```

## Product Boundary

v0.0.2 is the first playable Charterfall slice, not the full Constitutional Survival City Builder.

It should use the Milestone 0 artifacts as design inputs:

- [Charterfall pitch](../../design/charterfall-pitch.md)
- [Clause catalog](../../design/clauses.md)
- [First content pack](../../design/first-content-pack.md)
- [Player-facing metrics](../../design/player-facing-metrics.md)
- [Win, fail, and carryover rules](../../design/win-fail-carryover.md)
- [Framing and claims boundary](../../design/framing-boundaries.md)
- [First prototype run contract](../prototype-contract.md)

PolityKit remains the deterministic simulation layer. Charterfall is responsible for player flow, presentation, campaign state, clause selection, inquiry screens, comparison language, and fiction.

## Target Artifact

Create a playable prototype in the repository using the app structure chosen for Charterfall implementation.

Recommended target if no Charterfall app exists yet:

```text
src/Charterfall.App
```

The app may be a lightweight web app, desktop UI, or other local prototype shell, but it must expose the full playable loop without requiring the player to inspect raw JSON.

## Prototype Flow

The v0.0.2 flow is:

```text
Select settlement -> select crisis -> draft charter -> run crisis -> inspect inquiry -> amend charter -> rerun -> compare -> advance
```

### 1. Select Settlement

Use the first content pack settlement:

```text
Greywater Compact
```

The prototype may hard-code this settlement for v0.0.2. The UI should show enough settlement context to explain why local institutions matter: intermittent isolation, limited capacity, faction pressure, scarce resources, and unreliable outside help.

### 2. Select Crisis

Support the three first-run crises from the content pack:

| Order | Crisis | PolityKit source | Seed | Ticks |
|---:|---|---|---:|---:|
| 1 | Failed Harvest | `village-food-crisis` or `examples/village-food-crisis.json` | `12345` | `120` |
| 2 | Fever Season | `examples/medicine-shortage.json` | `24680` | `90` |
| 3 | Supply Office Scandal | `examples/corruption-stress.json` | `98765` | `80` |

The first implementation may force this order. Scenario selection can be a visible step even if only the next campaign crisis is enabled.

### 3. Draft Charter

Let the player select a small set of clauses from the Milestone 0 clause catalog.

Minimum required dimensions:

- Allocation method.
- Decision authority.
- Transparency.
- Accountability.
- Emergency powers.

The prototype can start with one simulation-authoritative allocation model plus game-layer-only clauses for dimensions that do not yet have direct PolityKit inputs.

Required handling:

- `run-model` clauses can be sent to PolityKit.
- `preset-dimension` clauses can be shown as player-facing choices only when backed by a selected preset.
- `game-layer-only` clauses must not be sent as authoritative simulation parameters.
- The UI must distinguish simulation-active effects from presentation-only effects.

### 4. Run Crisis

Create a PolityKit run using the first prototype run contract.

Minimum operation:

```text
POST /api/runs
```

The request must include:

- selected scenario or scenario file
- selected model or preset
- seed
- tick count
- any supported model parameters
- Charterfall metadata for settlement, campaign crisis order, selected clauses, and game-layer-only clauses

The prototype should preserve the submitted run configuration so same-seed reruns and comparisons are reproducible.

### 5. Inspect Inquiry

Show an inquiry dashboard after each crisis.

Minimum data surfaces:

- five player-facing metrics
- event timeline
- severe failures or harmed groups when available
- administrative load and bottlenecks
- trust or legitimacy movement
- collapse or warning signals
- selected clause effects and tradeoffs

The dashboard should use:

```text
GET /api/runs/{id}/dashboard
```

The inquiry screen should make outcomes understandable without raw JSON. It may include simple authored interpretation, but deterministic PolityKit metrics and events remain the source of truth.

### 6. Amend Charter

After inquiry, allow the player to change at least one clause or model-backed parameter before rerunning.

Minimum amendment support:

- switch between at least two allocation methods or presets
- preserve the same settlement, crisis, seed, and tick count
- keep game-layer-only clauses visible in campaign state
- show a short explanation of what changed before rerun

### 7. Rerun

Use a same-seed rerun operation for the amended charter.

Minimum operation:

```text
POST /api/runs/{id}/rerun
```

The amended run should be linked to the original run so the comparison view can explain what changed.

### 8. Compare

Show a before/after comparison after the amended rerun.

Minimum operation:

```text
GET /api/runs/{id}/compare/{comparisonId}
```

The comparison must include:

- metric deltas
- changed event or harm summary when available
- changed administrative load
- changed trust or legitimacy
- changed severe failures
- language that says results changed under the selected assumptions, not that a real-world system is proven better

### 9. Advance Campaign

After comparison, allow the player to continue to the next crisis.

Minimum carryover state:

- prior run IDs
- selected charter clauses
- compact inquiry summary
- win/fail/warning state
- limited power-incentive notes if used

The prototype can keep carryover lightweight. Do not build full save/load unless it is already cheap within the chosen app framework.

## UI Requirements

The prototype UI must include these screens or equivalent states:

| Screen | Purpose |
|---|---|
| Start / settlement | Present Greywater Compact and current campaign step. |
| Crisis forecast | Show the next crisis, known pressures, seed, and available resources or assumptions. |
| Charter draft | Let the player choose clauses and see tradeoffs. |
| Run progress / result | Submit the run and move into inquiry. |
| Public inquiry | Show metrics, events, harms, warnings, and clause consequences. |
| Amendment | Let the player revise at least one clause. |
| Comparison | Show original versus amended outcome. |
| Campaign outcome | Show win, fail, or continue state after each crisis and final outcome after the third crisis. |

The UI can be plain. It should prioritize legibility, repeatability, and completing the loop over visual polish.

## Technical Requirements

### API Integration

The prototype should integrate with these PolityKit surfaces:

- `GET /api/models`
- `GET /api/metrics`
- `GET /api/scenarios`
- `POST /api/runs`
- `GET /api/runs/{id}/dashboard`
- `POST /api/runs/{id}/rerun`
- `GET /api/runs/{id}/compare/{comparisonId}`

If one of these surfaces is not implemented or not yet suitable, add a small compatibility layer or documented fallback rather than changing the Charterfall flow.

### State Model

Maintain a Charterfall session state with:

- settlement ID
- campaign crisis index
- selected scenario
- selected seed
- selected tick count
- selected clause IDs
- simulation-active model or preset ID
- game-layer-only clause IDs
- current run ID
- amended run ID
- comparison ID or comparison result
- inquiry summaries per crisis
- win/fail/warning state

### Determinism

The same crisis, seed, tick count, model, parameters, and scenario input should produce the same result.

The prototype must not mutate deterministic run outputs in the game layer. It can annotate and present them.

### Error Handling

The prototype should show readable errors for:

- missing API service
- missing scenario
- unsupported model or preset
- invalid clause mapping
- failed run creation
- missing dashboard data
- failed rerun or comparison

Errors should tell the developer or player what to fix next.

## Claims And Framing Boundary

Include the required Charterfall claims boundary in UI copy or about/help text:

```text
Charterfall shows how fictional institutional rules behave inside declared simulation assumptions. It does not prove that a real-world political, economic, or social system is superior.
```

Comparison copy should prefer:

- "under these assumptions"
- "in this run"
- "with this scenario and seed"
- "the amended charter changed..."

Avoid:

- "this system is better"
- "this proves..."
- "real societies should..."
- ranking real-world ideologies, nations, communities, or institutions

## Out Of Scope

v0.0.2 should not build:

- full spatial city construction
- full district simulation
- polished visual art direction
- procedural narrative campaigns
- arbitrary timeline branching
- stress test mode
- sensitivity report view
- multiplayer
- scenario authoring tools
- historical or real-world scenario simulation
- AI-authored citizen stories as authoritative results
- a universal score proving one institution is best

These belong to later milestones unless needed as tiny internal scaffolding for the prototype.

## Acceptance Criteria

v0.0.2 is complete when:

1. A player can open the prototype and start a Charterfall session.
2. The player can see the Greywater Compact settlement context.
3. The player can run the first crisis from the campaign script.
4. The player can select a charter clause or preset that affects the PolityKit run.
5. The player can inspect a dashboard with the five player-facing metrics.
6. The player can inspect an event timeline or equivalent event summary.
7. The player can amend the charter after inquiry.
8. The player can rerun the same crisis with the same seed and amended charter.
9. The player can compare the original and amended outcomes.
10. The player can advance through all three scripted crises.
11. The prototype shows win, fail, or warning state using documented rules.
12. The prototype keeps game-layer-only clauses out of authoritative PolityKit inputs.
13. The prototype includes assumption-bound comparison and claims-boundary language.
14. Outcomes are understandable without reading raw JSON.

## Verification

Manual and smoke-test verification are both required.

Manual verification:

1. Start the PolityKit API service.
2. Start the Charterfall prototype.
3. Begin a Greywater Compact session.
4. Draft a charter.
5. Run Failed Harvest.
6. Inspect the inquiry dashboard.
7. Amend the charter.
8. Rerun Failed Harvest with the same seed.
9. Compare original and amended outcomes.
10. Continue through Fever Season and Supply Office Scandal.
11. Confirm final win, fail, or warning state appears.
12. Confirm no screen requires raw JSON to understand the result.

Smoke-test verification:

1. Confirm the app builds.
2. Confirm the app can reach the PolityKit API.
3. Confirm `POST /api/runs` succeeds for the first crisis.
4. Confirm dashboard data loads for the created run.
5. Confirm rerun and comparison operations succeed.
6. Confirm game-layer-only clauses are not sent as model parameters.
7. Confirm the claims-boundary sentence appears exactly once in player-facing static copy or help/about copy.

No stress, sensitivity, AI, or timeline-branch verification is required for v0.0.2.
