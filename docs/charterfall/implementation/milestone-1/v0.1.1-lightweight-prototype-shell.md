# Charterfall v0.1.1: Lightweight Prototype Shell

## Source

This implementation slice covers section `1. Lightweight Prototype Shell` from [Milestone 1: Prototype Product Framing](milestone-1-product-framing.md).

Milestone 1 creates the first playable Charterfall prototype. This slice creates the app shell that later sections can fill with scenario selection, charter clauses, API integration, inquiry, amendment, comparison, and campaign progression.

## Goal

Create a small Charterfall app surface that opens directly into the playable civic prototype and gives later implementation slices a stable place to attach the rest of the loop.

By the end of this slice:

- A contributor can launch the Charterfall prototype locally.
- The first screen is the playable Draft view, not a landing page.
- The app has visible navigation or routed states for Draft, Inquiry, Amendment, Comparison, and Final Outcome.
- The app owns campaign session state separately from PolityKit deterministic run outputs.
- The shell can show run history, loading states, empty states, and error states before full API wiring exists.

## Product Boundary

This slice builds the Charterfall game-layer shell. It should not move simulation logic into Charterfall.

| Responsibility | Owner |
|---|---|
| Deterministic simulation runs, scenarios, models, metrics, events, reruns, and comparisons | PolityKit |
| Player flow, presentation, campaign session state, selected clause IDs, run history display, and fictional framing | Charterfall |
| Citizen stories, faction reactions, tutorial text, and rich game feel | Later Charterfall slices |

The shell can call existing PolityKit APIs later, but this slice can begin with typed placeholder state and mockable service boundaries so the UI flow exists before every data integration is complete.

## Shell Flow

The shell should represent this full loop even before every state is fully functional:

```text
Draft -> Resolve -> Inquiry -> Amend -> Compare -> Advance -> Final Outcome
```

Minimum routed or view states:

| State | Route or view | Purpose |
|---|---|---|
| Draft | `/` or `/draft` | Start the playable loop with settlement, crisis, and charter placeholders. |
| Inquiry | `/inquiry` or in-app step | Show where dashboard metrics and events will appear. |
| Amendment | `/amend` or in-app step | Show where charter changes will be made after inquiry. |
| Comparison | `/compare` or in-app step | Show where original and amended runs will be compared. |
| Final Outcome | `/outcome` or in-app step | Show campaign result, win/fail state, and next-step summary. |

The player should never need to inspect raw JSON, run CLI commands, or edit files to move through the shell.

## Session State

Create a Charterfall-owned session model that can survive navigation during one local play session.

Minimum state shape:

| Field | Purpose | First value |
|---|---|---|
| `settlementId` | Selected fictional settlement. | `greywater-compact` |
| `chapterNumber` | Current crisis chapter. | `1` |
| `activeCrisisId` | Current crisis card. | `failed-harvest` |
| `selectedClauseIds` | Player-facing charter selections. | Empty or default allocation clause. |
| `authoritativeRunIds` | PolityKit run IDs returned by API calls. | Empty list. |
| `selectedContinuationRunId` | Run chosen to continue into the next chapter. | `null` |
| `compactInquirySummary` | Short player-facing outcome summary. | Empty placeholder. |
| `lastError` | Last shell/API error shown to player. | `null` |
| `isBusy` | Loading or resolving state. | `false` |

The session model should distinguish:

- **Authoritative PolityKit data**: run IDs, scenario IDs, model IDs, parameters, metrics, events, and comparison outputs.
- **Charterfall game-layer data**: selected clause IDs, chapter progression, fictional summaries, presentation-only choices, and UI state.

## Required UI Regions

The shell should provide stable regions that later slices can fill.

| Region | Required content |
|---|---|
| Header | Product name, current chapter, active crisis, and run status. |
| Main view | Current playable state: Draft, Inquiry, Amendment, Comparison, or Outcome. |
| Assumptions panel | Settlement, scenario, seed, ticks, model/preset placeholder, and claims-boundary language. |
| Run history panel | Original and amended run IDs when available; empty state before runs exist. |
| Action bar | Primary action for current state, secondary back/continue action where useful. |
| Status area | Loading, error, and empty-state messages. |

The shell should feel like a compact operational tool for repeated play. Avoid a marketing-style hero page; the first viewport should signal that the player is already drafting a civic response.

## Service Boundaries

Create service interfaces or equivalent boundaries before hard-coding HTTP calls throughout UI components.

Recommended boundaries:

| Service | Responsibility |
|---|---|
| `ICharterfallSessionStore` | Holds current campaign/session state. |
| `IPolityKitRunClient` | Later calls `POST /api/runs`, dashboard, rerun, and compare endpoints. |
| `IPrototypeContentProvider` | Provides Greywater Compact, crisis cards, default clauses, and metric labels. |
| `IPlayerMessageFormatter` | Formats assumption-bound player copy and error messages. |

The first slice may implement these with in-memory data. Later slices can replace or extend them without changing the player-facing shell.

## Loading, Empty, And Error States

The shell should include professional states before deep integration work begins.

Minimum states:

- **Loading**: resolving a run, loading content, or waiting for API response.
- **Empty run history**: no runs have been created yet.
- **Empty inquiry**: no dashboard response exists yet.
- **Unavailable API**: PolityKit API cannot be reached.
- **Invalid transition**: player tries to compare before an amended run exists.
- **Integration pending**: placeholder content is visible but not yet backed by deterministic output.

Errors should be player-readable and contributor-useful. For example:

```text
No PolityKit run has been created yet. Draft a charter and resolve the crisis before opening the inquiry.
```

## Navigation Rules

Initial navigation can be linear, with guarded transitions:

| From | To | Required condition |
|---|---|---|
| Draft | Inquiry | A run exists, or the app is in explicit placeholder mode. |
| Inquiry | Amendment | Inquiry screen has been viewed. |
| Amendment | Comparison | An amended run exists, or comparison is explicitly marked pending integration. |
| Comparison | Advance | Player chooses a continuation run or accepts the current run. |
| Advance | Final Outcome | Chapter 3 is complete. |

For this shell slice, it is acceptable for buttons to move between placeholder states if they clearly state that deterministic run integration is not wired yet.

## Implementation Steps

1. Create `src/Charterfall.App`.
2. Add the app project to `PolityKit.slnx`.
3. Create the app shell layout and first playable Draft view.
4. Add route or state navigation for Draft, Inquiry, Amendment, Comparison, and Final Outcome.
5. Add in-memory Charterfall session state.
6. Add placeholder content provider for Greywater Compact, Failed Harvest, default clause, and five metric labels.
7. Add run history panel with empty and populated states.
8. Add loading, error, invalid-transition, and integration-pending messages.
9. Add launch instructions near the app or in the Milestone 1 docs.
10. Add a lightweight verification path that confirms the app launches and the shell states are reachable.

## Verification

Launch instructions for the implemented shell live in [src/Charterfall.App/README.md](../../../../src/Charterfall.App/README.md).

Minimum manual verification:

1. Launch the app locally.
2. Confirm the first screen is Draft and shows Greywater Compact.
3. Confirm the shell shows current chapter, crisis, assumptions, and claims-boundary copy.
4. Navigate to Inquiry, Amendment, Comparison, and Final Outcome through visible actions.
5. Confirm run history has a clear empty state.
6. Trigger or simulate an invalid transition and confirm a player-readable error appears.

Minimum automated verification, once the app project exists:

- Build the solution.
- Add at least one shell-level test or component test that verifies the initial Draft state.
- Add one session-state test that verifies selected chapter, crisis, and run IDs can be stored without touching PolityKit simulation services.

Recommended commands after implementation:

```powershell
dotnet build PolityKit.slnx
dotnet test PolityKit.slnx --no-build
```

## Non-Goals

This slice should not implement:

- real PolityKit run creation
- dashboard parsing
- rerun or comparison API integration
- full scenario selection
- full clause catalog editing
- citizen stories
- faction reactions
- tutorialized first run
- persistent save/load beyond local session state
- AI-generated text

Those belong to later Milestone 1 or Milestone 2 slices.

## Acceptance Checklist

This slice is complete when:

- `src/Charterfall.App` exists and launches locally.
- The first screen is the Draft state.
- Draft, Inquiry, Amendment, Comparison, and Final Outcome states are reachable.
- The shell has persistent local session state for the current chapter.
- Run history is visible and handles empty state cleanly.
- Loading, error, empty, and integration-pending states exist.
- Charterfall-owned state is separated from future PolityKit deterministic run outputs.
- Claims-boundary language appears in the assumptions or status area.
- The app has documented launch instructions.
