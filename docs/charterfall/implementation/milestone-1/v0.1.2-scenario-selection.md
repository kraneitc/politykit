# Charterfall v0.1.2: Scenario Selection

## Source

This implementation slice covers section `2. Scenario Selection` from [Milestone 1: Prototype Product Framing](milestone-1-product-framing.md).

It builds on [v0.1.1-lightweight-prototype-shell.md](v0.1.1-lightweight-prototype-shell.md) by replacing the shell's hard-coded crisis placeholder with a real Charterfall scenario selection surface grounded in the first content pack.

## Goal

Expose the first three Charterfall crisis scenarios as playable campaign chapters.

By the end of this slice:

- The Draft view shows Greywater Compact and the active crisis before the player resolves a run.
- The player can see scenario source, seed, tick count, and assumptions before submission.
- The app can lock the first campaign to a fixed chapter order while still making the active crisis explicit.
- Charterfall owns crisis-card presentation and campaign order; PolityKit remains the source for deterministic scenario execution.
- Missing or unavailable scenario data produces a player-readable error rather than a broken shell.

## Product Boundary

Scenario selection is a Charterfall content and presentation feature. It should choose which scenario will be submitted to PolityKit later; it should not reinterpret scenario mechanics or duplicate the simulation.

| Responsibility | Owner |
|---|---|
| Scenario execution, built-in scenario metadata, validation, seed and tick handling | PolityKit |
| Settlement premise, crisis card text, chapter order, design purpose, stressed charter dimensions, and locked/unlocked campaign state | Charterfall |
| Rich crisis art, faction reactions, citizen story hooks, and tutorialized crisis introductions | Later Charterfall slices |

## Scenario Set

Use the first content pack as the authoritative Charterfall scenario list for Milestone 1.

| Chapter | Crisis ID | Display name | PolityKit source | Seed | Ticks | Design purpose |
|---:|---|---|---|---:|---:|---|
| 1 | `failed-harvest` | Failed Harvest | `village-food-crisis` or `examples/village-food-crisis.json` | `12345` | `120` | Tests allocation pressure and visible unmet need. |
| 2 | `fever-season` | Fever Season | `examples/medicine-shortage.json` | `24680` | `90` | Tests vulnerability, triage, and trust. |
| 3 | `supply-office-scandal` | Supply Office Scandal | `examples/corruption-stress.json` | `98765` | `80` | Tests transparency, accountability, legitimacy, and power incentives. |

The first implementation may lock the sequence to chapter order:

```text
Failed Harvest -> Fever Season -> Supply Office Scandal
```

If the app allows free scenario selection during development, clearly label that as a prototype/debug mode so the campaign contract remains understandable.

## Data Model

Add a Charterfall-owned crisis card model. Suggested shape:

| Field | Purpose |
|---|---|
| `id` | Stable Charterfall crisis ID, such as `failed-harvest`. |
| `chapterNumber` | Campaign chapter order. |
| `displayName` | Player-facing crisis name. |
| `summary` | One or two sentences of fictional framing. |
| `polityKitScenario` | Built-in scenario slug or scenario file path to submit later. |
| `seed` | Fixed seed for deterministic comparison. |
| `ticks` | Fixed crisis duration. |
| `designPurpose` | What the scenario teaches or stresses. |
| `stressedClauseDimensions` | Player-facing dimensions likely to matter. |
| `isUnlocked` | Whether the current campaign can choose this crisis. |
| `isIntegrationAvailable` | Whether the PolityKit source can currently be resolved by the chosen run path. |

Store the selected crisis in the Charterfall session state:

| Session field | Update |
|---|---|
| `chapterNumber` | Set from selected crisis. |
| `activeCrisisId` | Set to selected crisis ID. |
| `scenarioSource` | Set to `polityKitScenario`. |
| `seed` | Set to crisis seed. |
| `ticks` | Set to crisis ticks. |
| `assumptionsSummary` | Include settlement, crisis, seed, ticks, and source. |

## Content Provider

Extend the shell's `IPrototypeContentProvider` or equivalent service with scenario content.

Recommended methods:

```csharp
IReadOnlyList<CrisisCard> GetCampaignCrises();
CrisisCard GetActiveCrisis(int chapterNumber);
CrisisCard? FindCrisis(string crisisId);
```

The provider should be in-memory for this slice. Later work can move content into JSON, embedded resources, or a dedicated Charterfall content package.

## PolityKit Scenario Availability

The current API exposes built-in scenarios through:

```text
GET /api/scenarios
```

The built-in catalog currently includes `village-food-crisis`. The first content pack also references `examples/medicine-shortage.json` and `examples/corruption-stress.json`.

For this slice, choose one of these implementation paths:

| Path | Use when | Tradeoff |
|---|---|---|
| Charterfall fixed content only | Building selection UI before run creation. | Fastest; chapters 2 and 3 can show as selected content even if not yet executable through the API. |
| Promote examples to built-in PolityKit scenarios | Milestone 1 wants all three crises executable through API scenario slugs. | Cleaner player flow; changes PolityKit scenario catalog. |
| Add a Charterfall scenario resolver or content file loader | The app should submit example paths or authored Charterfall scenario files. | More flexible; requires careful file/path and validation boundaries. |

Recommended for v0.1.2:

- Implement fixed Charterfall crisis cards in the app content provider.
- Mark each crisis with `isIntegrationAvailable`.
- Treat `failed-harvest` as the first executable path because it can map to `village-food-crisis`.
- Leave promotion or loading of chapters 2 and 3 to the run-creation slice unless the API already supports the required source safely.

## Draft View Requirements

Update the Draft view so the selected scenario is visible before any run is created.

Required content:

- Settlement name: `Greywater Compact`.
- Active crisis name and chapter number.
- Crisis summary.
- Scenario source.
- Fixed seed.
- Tick count.
- Stressed charter dimensions.
- Integration status: available, pending, or unavailable.
- Claims-boundary or assumptions-boundary copy.

The Draft view should make clear that these are declared simulation assumptions. Suggested copy:

```text
This crisis will be tested under the listed scenario, seed, and tick assumptions.
```

## Scenario Selection UI

Minimum UI behavior:

- Show all three crisis cards in order.
- Highlight the active crisis.
- Lock future chapters until the campaign advances, unless debug/free-select mode is enabled.
- Allow selecting the current unlocked crisis.
- Show a disabled or pending state for crises whose PolityKit source is not yet executable.
- Update session state when the player selects a crisis.

Card content should be compact and scannable:

| Card field | Example |
|---|---|
| Chapter | `Chapter 1` |
| Crisis | `Failed Harvest` |
| Pressure | `Food scarcity and unmet need` |
| Source | `village-food-crisis` |
| Seed/ticks | `Seed 12345, 120 ticks` |
| Stressed dimensions | `Allocation, transparency, administrative load, emergency powers` |

## Assumptions Panel

The assumptions panel should show scenario data in a stable shape that later run creation can reuse.

Minimum display:

```text
Settlement: Greywater Compact
Crisis: Failed Harvest
Scenario: village-food-crisis
Seed: 12345
Ticks: 120
Chapter: 1 of 3
```

If a source is not yet executable, show:

```text
Integration pending: this crisis card is available for campaign planning, but run creation is not wired for this scenario yet.
```

## Error And Empty States

Add player-readable states for scenario selection:

- No crisis cards configured.
- Active chapter has no matching crisis card.
- Selected crisis is locked.
- Selected crisis has no scenario source.
- PolityKit scenario metadata is unavailable.
- Scenario exists in Charterfall content but cannot yet be executed by the selected run path.

Example error:

```text
Fever Season is part of the campaign, but its PolityKit scenario source is not executable through the current prototype path yet.
```

## Implementation Steps

1. Add a `CrisisCard` or equivalent model to `src/Charterfall.App`.
2. Extend the content provider with the three fixed campaign crisis cards.
3. Add scenario-related fields to the session state if they do not already exist.
4. Update the Draft view to show settlement and active crisis details.
5. Add a scenario card list or chapter selector.
6. Add locked, active, selected, and integration-pending visual states.
7. Update the assumptions panel with scenario, seed, ticks, and chapter count.
8. Add empty/error states for unavailable scenario content.
9. Optionally call `GET /api/scenarios` to display API availability, but keep the UI functional when the API is offline.
10. Add tests for crisis-card content and active chapter selection.

## Verification

Minimum manual verification:

1. Launch the Charterfall app.
2. Confirm the Draft view shows Greywater Compact.
3. Confirm all three crisis cards appear in chapter order.
4. Confirm Chapter 1 is active by default.
5. Confirm Failed Harvest shows `village-food-crisis`, seed `12345`, and `120` ticks.
6. Confirm Fever Season and Supply Office Scandal show their example sources, seeds, ticks, and integration status.
7. Confirm selecting a locked future crisis is blocked or clearly marked as debug-only.
8. Confirm the assumptions panel updates when the active crisis changes.

Minimum automated verification:

- Content-provider test returns exactly three campaign crisis cards in the documented order.
- Active-crisis lookup returns Failed Harvest for chapter `1`.
- Session-state test stores selected crisis ID, source, seed, and ticks.
- Draft-view or component test renders the active crisis name and assumptions.

Recommended commands after implementation:

```powershell
dotnet build PolityKit.slnx
dotnet test PolityKit.slnx --no-build
```

## Non-Goals

This slice should not implement:

- PolityKit run creation
- dashboard inquiry
- rerun or comparison
- full freeform scenario authoring
- procedural campaign generation
- scenario balancing
- citizen story generation
- AI scenario suggestions
- promotion of example scenarios into built-in PolityKit scenarios unless chosen explicitly during implementation

## Acceptance Checklist

This slice is complete when:

- The Draft view displays the selected settlement and active crisis before running.
- All three first-pack crises are represented as Charterfall crisis cards.
- Scenario source, seed, tick count, chapter number, and assumptions are visible before submission.
- The app can lock campaign order while still naming the active crisis explicitly.
- Future or unavailable crises have clear disabled, locked, or integration-pending states.
- Scenario selection updates Charterfall session state.
- PolityKit scenario execution remains a later API-integration concern unless a source is already available.
- No crisis copy implies a real-world place, community, or institution is being judged.
