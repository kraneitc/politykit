# Charterfall v0.1.3: Charter Clause Selection

## Source

This implementation slice covers section `3. Charter Clause Selection` from [Milestone 1: Prototype Product Framing](milestone-1-product-framing.md).

It builds on [v0.1.1-lightweight-prototype-shell.md](v0.1.1-lightweight-prototype-shell.md) and [v0.1.2-scenario-selection.md](v0.1.2-scenario-selection.md) by replacing placeholder charter state with a player-facing clause selection surface grounded in the clause catalog.

## Goal

Let the player draft a small charter before the first crisis and preserve that selection for later run creation, inquiry, amendment, and comparison.

By the end of this slice:

- The Draft view shows selectable charter clauses with stable IDs, player-facing text, and tradeoffs.
- At least one selected clause maps to an authoritative PolityKit run model or approved parameter.
- Non-authoritative clauses are clearly marked and stored as Charterfall game-layer state.
- The app can derive a future run configuration from selected authoritative clauses without sending `game-layer-only` choices to PolityKit.
- The player can understand what each clause is expected to affect before resolving a crisis.

## Product Boundary

Charter clause selection is a Charterfall game-layer feature that prepares deterministic run inputs. PolityKit remains responsible for executing mapped models, presets, parameters, scenarios, metrics, events, reruns, and comparisons.

| Responsibility | Owner |
|---|---|
| Player-facing clause cards, dimensions, tradeoff text, selected IDs, presentation-only effects, and amendment UI state | Charterfall |
| Model IDs, preset IDs, supported parameters, deterministic execution, dashboard output, rerun, and comparison | PolityKit |
| Citizen stories, faction reactions, power-incentive carryover, and rich tutorialization | Later Charterfall slices |

Do not create UI choices that imply simulation behavior without either a documented PolityKit mapping or an explicit `game-layer-only` label.

## Clause Surface

Use [clauses.md](../../design/clauses.md) as the source of truth for Milestone 1 clause content.

Initial dimensions:

| Dimension | Required in Draft? | First playable options | Prototype behavior |
|---|---:|---|---|
| Allocation method | Yes | Need-based, market-based, hierarchy-based, participatory, hybrid | At least one option must map to a `run-model` or preset. |
| Decision authority | Optional in first pass | Council, expert office, local districts, emergency executive | May be shown as `preset-dimension` or `game-layer-only`. |
| Transparency | Optional in first pass | Public ledger, delayed reporting, closed administration | May be shown as `preset-dimension` or `game-layer-only`. |
| Accountability | Optional in first pass | Appeal board, audit office, citizen review, none | May be shown as `preset-dimension` or `game-layer-only`. |
| Emergency powers | Optional in first pass | None, limited, renewable, broad | `game-layer-only` until explicit model inputs exist. |

Recommended v0.1.3 player surface:

- Make `Allocation method` the required simulation-active choice.
- Show one or two additional dimensions as clearly labeled presentation or preset-backed choices.
- Defer free mixing of all dimensions until composition rules exist.

## Clause Data Model

Add a Charterfall-owned clause definition model. Suggested shape:

| Field | Purpose |
|---|---|
| `id` | Stable clause ID, such as `allocation.need_based`. |
| `dimension` | Clause group, such as `allocation_method`. |
| `displayName` | Player-facing name. |
| `description` | Short explanation of the clause. |
| `tradeoff` | What the player gains and risks. |
| `mappingStatus` | `run-model`, `preset-dimension`, or `game-layer-only`. |
| `modelId` | PolityKit model or preset ID when authoritative. |
| `parameters` | Approved parameter defaults when authoritative. |
| `gameLayerEffects` | Presentation, badges, copy, or campaign notes. |
| `availability` | First run, deferred, debug-only, or later. |
| `boundary` | Fictional and assumption-bound wording. |
| `powerIncentiveTags` | Optional tags such as `precedent`, `opacity`, `capture`, or `accountability-debt`. |

The model should be local to Charterfall until the catalog becomes machine-readable.

## Selection Rules

Enforce simple rules before API integration:

| Rule | Reason |
|---|---|
| Exactly one allocation clause is selected. | First authoritative simulation choice must be unambiguous. |
| Only one clause per dimension unless the dimension supports multi-select. | Prevent contradictory charter state. |
| At least one selected clause must be authoritative. | Run creation needs a model or preset path. |
| `game-layer-only` clauses remain selected in Charterfall state but do not enter run request drafts. | Preserve simulation boundary. |
| `preset-dimension` clauses are explanatory unless selected through an authored preset. | Avoid fake composition. |
| Deferred clauses may appear disabled with explanation, not as active choices. | Preserve player trust. |

Suggested defaults:

```text
allocation.need_based
```

If optional dimensions are shown, use safe defaults:

```text
transparency.public_ledger
accountability.appeal_board
emergency.none
```

Mark non-authoritative defaults clearly.

## Session State

Extend the Charterfall session state from v0.1.1.

| Field | Purpose |
|---|---|
| `selectedClauseIds` | Full player-facing charter selection. |
| `authoritativeModelIds` | Derived PolityKit model or preset IDs for future run creation. |
| `authoritativeParameters` | Derived approved parameters for future run creation. |
| `gameLayerClauseIds` | Selected clauses that affect presentation or campaign state only. |
| `clauseSelectionErrors` | Validation messages such as no authoritative clause selected. |
| `charterSummary` | Short player-facing summary for assumptions panel and inquiry carryover. |

The session should store selected clause IDs as the durable source. Derived model IDs and parameters can be recalculated from clause definitions.

## Run Input Preview

Before API integration, the Draft view should show a read-only run input preview generated from selected clauses.

Example:

```json
{
  "models": [
    "need-based-allocation"
  ],
  "parameters": {
    "needPriorityWeight": 1.0,
    "vulnerabilityPriorityWeight": 0.5
  },
  "gameLayerOnlyClauses": [
    "emergency.limited"
  ]
}
```

This is a preview, not a submitted request. Later v0.1.4 run creation should combine it with selected scenario, seed, and tick count.

## UI Requirements

Update the Draft view to include a charter drafting area.

Required content:

- Clause dimensions as grouped controls.
- Clause cards or segmented controls with display name, description, and tradeoff.
- Mapping status indicator: simulation-active, preset-backed, or presentation-only.
- Disabled/deferred state where a catalog option is not available.
- Selected charter summary.
- Run input preview for authoritative model IDs and parameters.
- Boundary copy explaining that clauses are fictional and assumption-bound.

Recommended copy for `game-layer-only` clauses:

```text
This clause affects Charterfall presentation and campaign notes in this prototype. It is not yet a PolityKit simulation input.
```

## Validation

Validate clause selection before the player can resolve a crisis.

Minimum validation cases:

| Case | Result |
|---|---|
| No allocation clause selected | Block run and ask the player to choose allocation. |
| More than one allocation clause selected | Block run and ask the player to choose one. |
| Only `game-layer-only` clauses selected | Block run and explain that at least one simulation-active clause is required. |
| `preset-dimension` selected without preset context | Allow as explanatory only or block if the UI implies it is active. |
| Unknown clause ID in session state | Remove it or mark it invalid with a recoverable error. |
| Deferred clause selected through stale state | Mark unavailable and ask player to revise. |

Errors should be visible in the status area and near the clause group.

## Content Provider

Extend `IPrototypeContentProvider` or equivalent service with clause content.

Recommended methods:

```csharp
IReadOnlyList<CharterClauseDefinition> GetClauses();
IReadOnlyList<CharterClauseDefinition> GetClausesForDimension(string dimension);
CharterClauseDefinition? FindClause(string clauseId);
CharterRunInputPreview BuildRunInputPreview(IReadOnlyList<string> selectedClauseIds);
ClauseSelectionValidationResult ValidateClauseSelection(IReadOnlyList<string> selectedClauseIds);
```

If this logic grows, split run-input derivation into a dedicated service such as `ICharterClauseMapper`.

## Amendment Readiness

This slice primarily supports Draft, but it should leave room for the later Amendment state.

Design now for:

- reusing the same clause selection component in Amendment
- showing original selected clauses beside amended clauses later
- computing changed clause IDs
- preserving same-seed assumptions after amendment
- preventing `game-layer-only` amendments from being presented as deterministic run changes

Do not implement full amendment comparison yet.

## Implementation Steps

1. Add a `CharterClauseDefinition` model to `src/Charterfall.App`.
2. Add mapping status values for `run-model`, `preset-dimension`, and `game-layer-only`.
3. Extend the content provider with the minimum allocation clauses and optional presentation clauses.
4. Add selected clause state to the session store.
5. Build grouped clause selection UI in the Draft view.
6. Add mapping status labels and explanatory copy.
7. Add validation for required authoritative selection.
8. Add run input preview derived from selected clauses.
9. Update the assumptions panel with selected charter summary.
10. Add tests for content, validation, and run input preview generation.

## Verification

Minimum manual verification:

1. Launch the Charterfall app.
2. Confirm the Draft view shows at least the allocation clause group.
3. Confirm Need-Based, Market-Based, and Hierarchy-Based allocation options are selectable.
4. Confirm only one allocation option can be selected at a time.
5. Confirm selecting Need-Based previews `need-based-allocation` with default parameters.
6. Confirm selecting Market-Based previews `market-based-allocation`.
7. Confirm any `game-layer-only` clause is visibly marked and excluded from authoritative model/parameter preview.
8. Confirm the player cannot proceed with no authoritative clause selected.
9. Confirm the assumptions panel names the selected charter.

Minimum automated verification:

- Content-provider test returns the minimum allocation clauses with stable IDs.
- Validation test rejects no authoritative clause.
- Validation test rejects multiple allocation clauses.
- Mapping test derives `models: ["need-based-allocation"]` and default parameters for `allocation.need_based`.
- Mapping test excludes `game-layer-only` clauses from authoritative run input.
- Session-state test persists selected clause IDs independently from derived run input.

```powershell
dotnet test PolityKit.slnx --no-build
```

## Non-Goals

This slice should not implement:

- PolityKit run creation
- freeform clause composition
- full preset editing
- model parameter tuning UI beyond documented defaults
- amendment reruns
- before/after comparison
- power-incentive carryover
- citizen story changes from selected clauses
- AI-generated clause analysis

## Acceptance Checklist

This slice is complete when:

- The Draft view exposes selectable charter clauses.
- Every visible clause has a stable ID, display name, description, tradeoff, mapping status, and boundary text.
- At least one selected clause maps to an authoritative PolityKit model or approved parameter.
- `game-layer-only` clauses are stored in Charterfall session state but excluded from authoritative run input preview.
- The UI prevents unsupported clause composition from looking simulation-active.
- The session stores selected clause IDs for later run creation and amendment.
- The assumptions panel summarizes the selected charter.
- No clause copy implies real-world proof or endorsement.
