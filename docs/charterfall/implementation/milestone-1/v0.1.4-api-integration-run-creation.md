# Charterfall v0.1.4: API Integration For Run Creation

## Source

This implementation slice covers section `4. API Integration For Run Creation` from [Milestone 1: Prototype Product Framing](milestone-1-product-framing.md).

It builds on [v0.1.1-lightweight-prototype-shell.md](v0.1.1-lightweight-prototype-shell.md), [v0.1.2-scenario-selection.md](v0.1.2-scenario-selection.md), and [v0.1.3-charter-clause-selection.md](v0.1.3-charter-clause-selection.md) by wiring the Draft state to PolityKit's deterministic run-creation API.

## Goal

Let the player resolve the selected crisis under the selected charter assumptions by creating a PolityKit run from the Charterfall Draft view.

By the end of this slice:

- The Draft view can submit a run through `POST /api/runs`.
- The request is built only from selected crisis data and authoritative mapped charter clauses.
- The returned run ID is persisted in Charterfall session state.
- The submitted request is inspectable by contributors.
- The UI moves from Draft or Resolve into Inquiry readiness after successful run creation.
- API errors are shown as player-readable messages without corrupting campaign state.

## Product Boundary

Run creation is the first point where Charterfall hands declared assumptions to PolityKit.

| Responsibility | Owner |
|---|---|
| Selected settlement, chapter, crisis card, selected clause IDs, run history display, debug request preview, and player-facing errors | Charterfall |
| Scenario resolution, model and preset execution, parameter handling, deterministic simulation, persisted run ID, and run detail response | PolityKit |
| Dashboard interpretation, inquiry summaries, amendment reruns, comparison, and campaign carryover | Later Milestone 1 slices |

Charterfall must not send `game-layer-only` clauses, citizen fiction, faction reactions, power-incentive notes, or campaign summaries as PolityKit simulation inputs.

## API Contract

Use the existing PolityKit endpoint:

```http
POST /api/runs
```

Current request shape:

```csharp
public sealed class CreateRunRequest
{
    public string? Scenario { get; init; }
    public int? Seed { get; init; }
    public int? Ticks { get; init; }
    public IReadOnlyList<string>? Models { get; init; }
    public IReadOnlyDictionary<string, double>? Parameters { get; init; }
}
```

Current success response shape:

```csharp
public sealed class RunDetailResponse
{
    public Guid Id { get; init; }
    public DateTimeOffset CreatedAt { get; init; }
    public string ScenarioName { get; init; } = "";
    public int Seed { get; init; }
    public int Ticks { get; init; }
    public IReadOnlyList<ModelRunSummaryResponse> Models { get; init; } = [];
}
```

Invalid requests return `400 Bad Request` with `ProblemDetails` titled `Run request is invalid.`

## Request Composition

Build the run request from the selected scenario and clause-derived run input preview.

| Request field | Charterfall source | Rule |
|---|---|---|
| `scenario` | Active crisis card `polityKitScenario` | Use only executable PolityKit source. For first run, prefer `village-food-crisis`. |
| `seed` | Active crisis card `seed` | Preserve fixed content-pack seed, starting with `12345` for Failed Harvest. |
| `ticks` | Active crisis card `ticks` | Preserve fixed content-pack ticks, starting with `120` for Failed Harvest. |
| `models` | Authoritative clause mapper | Include only `run-model` model IDs or selected preset IDs. |
| `parameters` | Authoritative clause mapper | Include only approved mapped parameters. Omit or send empty object when none apply. |

Do not include:

- `settlementId`
- `chapterNumber`
- `selectedClauseIds`
- `gameLayerOnlyClauses`
- `charterSummary`
- `compactInquirySummary`
- citizen story data
- power-incentive notes

Those remain Charterfall session state.

## First Executable Path

Use this as the minimum reliable first run:

```json
{
  "scenario": "village-food-crisis",
  "models": [
    "need-based-allocation"
  ],
  "seed": 12345,
  "ticks": 120,
  "parameters": {
    "needPriorityWeight": 1.0,
    "vulnerabilityPriorityWeight": 0.5
  }
}
```

This combines:

- Chapter 1 from the first content pack: `failed-harvest`.
- Built-in PolityKit scenario: `village-food-crisis`.
- Default authoritative clause: `allocation.need_based`.
- Clause catalog parameter defaults: `needPriorityWeight=1.0`, `vulnerabilityPriorityWeight=0.5`.

Chapters 2 and 3 may remain integration-pending until their scenario sources are executable through the chosen run path.

## Client Boundary

Implement or extend `IPolityKitRunClient`.

Recommended methods:

```csharp
Task<CreateRunResult> CreateRunAsync(CreateRunInput input, CancellationToken cancellationToken);
```

Suggested app-owned input:

| Field | Purpose |
|---|---|
| `scenario` | PolityKit scenario slug or executable source. |
| `seed` | Fixed deterministic seed. |
| `ticks` | Crisis duration. |
| `models` | Authoritative model or preset IDs. |
| `parameters` | Approved numeric parameters. |

Suggested result:

| Field | Purpose |
|---|---|
| `id` | PolityKit run ID. |
| `createdAt` | Run creation timestamp. |
| `scenarioName` | Resolved scenario display name from PolityKit. |
| `seed` | Resolved seed. |
| `ticks` | Resolved tick count. |
| `models` | Resolved model summaries or names. |
| `rawRequest` | Serialized request for debug panel. |

The client should translate HTTP and `ProblemDetails` failures into player-readable app errors without exposing stack traces.

## Session State

Extend Charterfall session state with run-creation fields.

| Field | Purpose |
|---|---|
| `pendingRunRequest` | Last composed request before submission. |
| `lastSubmittedRunRequest` | Request that created the latest run. |
| `authoritativeRunIds` | Append successful run ID. |
| `currentRunId` | The run selected for the next Inquiry view. |
| `lastRunCreatedAt` | Timestamp for run history display. |
| `lastResolvedScenarioName` | PolityKit-resolved scenario name. |
| `lastRunError` | Last run-creation failure. |
| `isResolvingRun` | Loading state for the Resolve action. |

Do not clear selected crisis or selected clauses after a failed run. The player should be able to revise and retry.

## Draft View Requirements

Update the Draft view action from placeholder navigation to real run creation when the active crisis and selected charter are executable.

Required Draft content before submission:

- Settlement name.
- Active crisis and chapter.
- Scenario source.
- Seed and tick count.
- Selected charter summary.
- Authoritative model IDs.
- Approved parameters.
- Game-layer-only clauses shown separately.
- Claims-boundary or assumptions-boundary copy.
- Submit button labeled as resolving the crisis, not as generic save.

Suggested primary action:

```text
Resolve crisis
```

If the active crisis or charter is not executable, disable the action and explain what is pending.

## Resolve State

Add a short Resolve state or loading overlay.

Required behavior:

- Set `isResolvingRun=true`.
- Disable duplicate submissions.
- Show the scenario, model, seed, and ticks being submitted.
- On success, append the run ID and move to Inquiry readiness.
- On failure, stay on Draft and show a recoverable error.

The first implementation may route directly to Inquiry after success, even if the Inquiry screen still contains placeholder dashboard content. The run ID must be available to the later inquiry slice.

## Debug Request Panel

The submitted request must be inspectable through at least one of:

- visible debug panel in the Draft or Resolve state
- developer-only collapsible panel
- browser console log
- saved app session summary

Preferred for Milestone 1: a collapsible debug panel that shows:

```json
{
  "scenario": "village-food-crisis",
  "seed": 12345,
  "ticks": 120,
  "models": [
    "need-based-allocation"
  ],
  "parameters": {
    "needPriorityWeight": 1.0,
    "vulnerabilityPriorityWeight": 0.5
  }
}
```

Label it as a developer aid, not player lore.

## Error Handling

Handle these cases explicitly:

| Case | UI behavior |
|---|---|
| PolityKit API is offline | Show unavailable-service error and keep Draft state intact. |
| Active crisis is integration-pending | Block submission and explain that the scenario is not executable yet. |
| No authoritative model selected | Block submission and point back to charter selection. |
| PolityKit rejects scenario | Show `ProblemDetails.Detail` when available. |
| PolityKit rejects model or parameter | Show mapped error and keep selected clauses for correction. |
| Network timeout | Offer retry without duplicating run history. |
| Duplicate click while resolving | Ignore or disable second submission. |

Do not append a run ID unless the API returns a successful `RunDetailResponse`.

## Implementation Steps

1. Add app-owned request/result models for run creation.
2. Implement or extend `IPolityKitRunClient`.
3. Configure the PolityKit API base URL for local development.
4. Compose `CreateRunRequest` from active crisis and authoritative clause preview.
5. Validate executable scenario and authoritative clause selection before submission.
6. Add a Resolve action to the Draft view.
7. Add `isResolvingRun`, success, and error states.
8. Persist returned run ID in Charterfall session state.
9. Append successful runs to the run history panel.
10. Add a debug request panel for the submitted request.
11. Route or transition to Inquiry readiness after success.
12. Add tests for request composition, `game-layer-only` exclusion, success persistence, and failure handling.

## Verification

Minimum manual verification:

1. Start the PolityKit API locally.
2. Launch the Charterfall app.
3. Confirm Draft shows Failed Harvest, `village-food-crisis`, seed `12345`, and `120` ticks.
4. Select `allocation.need_based`.
5. Open the debug request preview and confirm it matches the expected request.
6. Click `Resolve crisis`.
7. Confirm duplicate submissions are disabled while the request is running.
8. Confirm the returned run ID appears in run history.
9. Confirm the app transitions to Inquiry readiness with the current run ID available.
10. Stop the API and confirm retryable unavailable-service messaging appears.

Minimum automated verification:

- Request-composition test includes scenario, seed, ticks, model IDs, and approved parameters.
- Request-composition test excludes `game-layer-only` clauses.
- Validation test blocks integration-pending crisis sources.
- Client test handles `ProblemDetails` errors.
- Session-state test persists successful run ID and does not mutate selected clauses after failure.
- UI/component test verifies the Resolve action is disabled when no authoritative model is selected.

Recommended commands after implementation:

```powershell
dotnet build PolityKit.slnx
dotnet test PolityKit.slnx --no-build
```

## Non-Goals

This slice should not implement:

- dashboard inquiry rendering
- metric translation
- rerun with amended clauses
- comparison view
- full campaign advancement
- executable support for every example-file scenario unless chosen explicitly
- freeform model parameter tuning
- AI-generated explanation or summary

## Acceptance Checklist

This slice is complete when:

- Draft can submit `POST /api/runs` for the first executable crisis.
- The submitted request includes scenario, seed, ticks, authoritative models, and approved parameters.
- `game-layer-only` clauses are excluded from the API request.
- The UI persists the returned run ID in Charterfall session state.
- Run history shows the created run.
- The submitted request is inspectable through a debug surface.
- API failures are recoverable and do not corrupt selected crisis or charter state.
- The app can transition from successful run creation into Inquiry readiness.
