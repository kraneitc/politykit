# Charterfall v0.1.5: Dashboard Inquiry View

## Source

This implementation slice covers section `5. Dashboard Inquiry View` from [Milestone 1: Prototype Product Framing](milestone-1-product-framing.md).

It builds on [v0.1.4-api-integration-run-creation.md](v0.1.4-api-integration-run-creation.md) by turning a created PolityKit run ID into the first playable public inquiry screen.

## Goal

Let the player understand what happened in a completed crisis without reading raw JSON.

By the end of this slice:

- The Inquiry view can load `GET /api/runs/{id}/dashboard` for the current run.
- The screen shows the five player-facing metrics: Needs Met, Severe Failures, Trust, Inequality, and Administrative Load.
- The screen shows an event timeline or event list.
- The screen shows severe-failure, collapse, or warning signals where available.
- The screen includes an assumptions block with scenario, seed, model, parameters, chapter, and selected charter summary.
- Inquiry copy remains fictional, assumption-bound, and traceable to deterministic PolityKit data.

## Product Boundary

The inquiry screen is Charterfall presentation over PolityKit deterministic outputs.

| Responsibility | Owner |
|---|---|
| Dashboard endpoint, raw metrics, raw events, run summary, scenario name, seed, ticks, model summaries | PolityKit |
| Player-facing metric labels, inquiry layout, event grouping, assumptions panel, compact explanation, and next-step prompts | Charterfall |
| Citizen story cards, faction testimony, AI summaries, and rich public hearing presentation | Later slices |

Do not invent authoritative harms, outcomes, citizen stories, or causal claims that are not grounded in dashboard data or documented game-layer state.

## API Contract

Use the existing PolityKit endpoint:

```http
GET /api/runs/{id}/dashboard
```

Current response shape:

```csharp
public sealed class RunDashboardResponse
{
    public Guid Id { get; init; }
    public DateTimeOffset CreatedAt { get; init; }
    public string ScenarioName { get; init; } = "";
    public int Seed { get; init; }
    public int Ticks { get; init; }
    public SimulationRunSummary Summary { get; init; } = new();
    public IReadOnlyList<MetricResponse> Metrics { get; init; } = [];
    public IReadOnlyList<EventResponse> Events { get; init; } = [];
}
```

Metric response:

```csharp
public sealed class MetricResponse
{
    public string Model { get; init; } = "";
    public int Tick { get; init; }
    public string Name { get; init; } = "";
    public double Value { get; init; }
    public string Unit { get; init; } = "";
}
```

Event response:

```csharp
public sealed class EventResponse
{
    public string Model { get; init; } = "";
    public int Tick { get; init; }
    public string Type { get; init; } = "";
    public string Description { get; init; } = "";
    public IReadOnlyDictionary<string, object> Data { get; init; } = new Dictionary<string, object>();
}
```

Missing runs return `404 Not Found`.

## Inquiry Data Model

Create a Charterfall-owned inquiry view model translated from the dashboard response.

Suggested shape:

| Field | Purpose |
|---|---|
| `runId` | Current authoritative PolityKit run ID. |
| `scenarioName` | PolityKit-resolved scenario display name. |
| `chapterNumber` | Charterfall campaign chapter. |
| `seed` | Run seed. |
| `ticks` | Run length. |
| `selectedClauseIds` | Charterfall-selected clauses used for the run. |
| `modelNames` | Models represented in dashboard output. |
| `metricCards` | Five player-facing metric cards. |
| `events` | Timeline or grouped event list. |
| `warningSignals` | Severe-failure, collapse, trust, or load warnings derived from metrics/events. |
| `compactSummary` | Short deterministic-data-grounded inquiry summary. |
| `rawDashboard` | Optional developer/debug payload. |

This view model should be derived from dashboard data and session state. It should not become a new simulation source of truth.

## Metric Mapping

Use [player-facing-metrics.md](../../design/player-facing-metrics.md) as the first inquiry metric map.

| Player-facing label | PolityKit metric name | Direction | First display rule |
|---|---|---|---|
| Needs Met | `Needs Met` | Higher is better | Show final value for current run. |
| Severe Failures | `Severe Failures` | Lower is better | Show final value and warning if above documented threshold when available. |
| Trust | `Trust` | Higher is better | Show final value and low-trust warning when available. |
| Inequality | `Inequality` | Lower is better | Show final value and mark as distributional tradeoff. |
| Administrative Load | `Administrative Load` | Lower is usually better | Show final value and high-load warning when available. |

For v0.1.5, use final metric values from either:

- `dashboard.Summary.Models[*].FinalMetrics`, if easiest to consume
- or the last `dashboard.Metrics` value per model and metric name

If multiple models are present in one run, group metrics by model or choose a clearly labeled primary model. Do not silently average model outputs in Milestone 1.

## Event Timeline

Show a compact event timeline sourced from `dashboard.Events`.

Minimum event fields:

- tick
- model
- event type
- description

Recommended UI behavior:

- Sort by tick ascending.
- Allow filtering by model when multiple models are present.
- Highlight event types that indicate severe failures, collapse, shortage, trust movement, administrative backlog, or allocation failures when those event types exist.
- Show an empty state when the dashboard has no events.

Do not rewrite event descriptions into stronger claims than the source supports.

## Warning Signals

Create first-pass warning signals from available metrics and event types.

Minimum signals:

| Signal | Source | First behavior |
|---|---|---|
| Low needs met | `Needs Met` final value | Warn if value falls below documented threshold when available, otherwise mark as watch item only. |
| Severe failures | `Severe Failures` final value or event type | Warn when present or elevated. |
| Low trust | `Trust` final value | Warn if value falls below documented threshold when available. |
| High administrative load | `Administrative Load` final value | Warn if value is high or if load-related events exist. |
| Collapse or near-collapse | Event type or summary signal where available | Show as prominent inquiry signal. |

If a threshold is not yet implemented in app code, label the signal as `watch` rather than `fail`.

## Assumptions Block

The Inquiry view must repeat the assumptions that shaped the run.

Minimum display:

```text
Scenario: Village Food Crisis
Chapter: 1 of 3
Seed: 12345
Ticks: 120
Model: NeedBasedAllocation
Charter: Need-Based Allocation
Parameters: needPriorityWeight=1.0, vulnerabilityPriorityWeight=0.5
```

Also show:

```text
These results describe this fictional run under the listed assumptions.
```

The assumptions block should combine PolityKit dashboard values with Charterfall session state from run creation. If they differ, prefer PolityKit values for run facts and show a debug warning.

## Inquiry Copy

The first inquiry summary should be deterministic-data-grounded and modest.

Allowed language:

- "In this run, Needs Met ended at..."
- "Severe Failures were recorded by the dashboard."
- "Trust changed under these assumptions."
- "The event timeline shows..."
- "This suggests an amendment to inspect, not proof of a real-world rule."

Avoid:

- "This charter is best."
- "This proves this institution works."
- "This citizen suffered" unless a structured event or later story-card source supports it.
- real-world political, economic, or social ranking.

## Client Boundary

Extend `IPolityKitRunClient` or add an inquiry-specific client method.

Recommended method:

```csharp
Task<RunDashboardResult> GetDashboardAsync(Guid runId, CancellationToken cancellationToken);
```

Create a mapper such as:

```csharp
InquiryViewModel BuildInquiry(RunDashboardResult dashboard, CharterfallSessionState session);
```

Keep API DTOs separate from player-facing view models so UI language can evolve without changing the PolityKit contract.

## Session State

Extend Charterfall session state with inquiry fields.

| Field | Purpose |
|---|---|
| `currentDashboardRunId` | Run ID used for latest inquiry load. |
| `lastDashboardLoadedAt` | Timestamp for refresh/debug display. |
| `inquiryMetricSnapshot` | Five displayed metric values. |
| `inquiryEventCount` | Number of events loaded. |
| `inquiryWarningSignals` | Derived warning/watch labels. |
| `compactInquirySummary` | Short summary carried into amendment and campaign state. |
| `lastInquiryError` | Last dashboard load failure. |
| `isLoadingInquiry` | Loading state for dashboard fetch. |

Do not mutate selected clauses or run IDs when dashboard loading fails.

## UI Requirements

Required Inquiry view regions:

| Region | Required content |
|---|---|
| Inquiry header | Crisis name, chapter, run ID, and model. |
| Metric strip | Five player-facing metric cards. |
| Warning panel | Collapse, severe-failure, trust, load, or watch signals. |
| Event timeline | Tick-ordered event list. |
| Assumptions block | Scenario, seed, ticks, model, parameters, charter, and boundary copy. |
| Debug panel | Raw metric names, dashboard fetch status, and optional raw payload. |
| Action bar | Continue to Amendment; return to Draft if run is missing. |

The Inquiry view should be the first place the prototype starts to feel like a civic review rather than an API dashboard.

## Loading, Empty, And Error States

Handle these cases:

| Case | UI behavior |
|---|---|
| No current run ID | Show empty inquiry and route player back to Draft/Resolve. |
| Dashboard loading | Show run ID and loading state. |
| Dashboard 404 | Show missing-run error and keep run history intact. |
| API offline | Show retryable unavailable-service error. |
| Missing metric | Show metric card as unavailable and record debug warning. |
| No events | Show a useful empty timeline state. |
| Multiple models | Group or clearly label metrics/events by model. |

## Implementation Steps

1. Add dashboard client method to `IPolityKitRunClient`.
2. Add app-owned dashboard result models or API DTOs if not already shared.
3. Add an inquiry mapper from `RunDashboardResponse` to a Charterfall inquiry view model.
4. Add the five metric definitions from `player-facing-metrics.md`.
5. Update the Inquiry route/view to load the current run dashboard.
6. Render five metric cards with player-facing labels and raw metric traceability.
7. Render an event timeline from dashboard events.
8. Render assumptions block using dashboard facts plus session state.
9. Add warning/watch signal derivation.
10. Add loading, empty, and error states.
11. Persist compact inquiry summary in session state.
12. Add tests for dashboard mapping, missing metrics, event sorting, and assumptions display.

## Verification

Minimum manual verification:

1. Start the PolityKit API locally.
2. Launch the Charterfall app.
3. Create a run through Draft using Failed Harvest and Need-Based Allocation.
4. Confirm the app transitions to Inquiry.
5. Confirm Inquiry loads `GET /api/runs/{id}/dashboard`.
6. Confirm all five player-facing metric cards appear.
7. Confirm raw metric names remain traceable in debug or tooltip text.
8. Confirm the event timeline is populated or shows a clean empty state.
9. Confirm assumptions show scenario, seed, ticks, model, parameters, chapter, and charter.
10. Stop the API and confirm the Inquiry view shows a retryable error.

Minimum automated verification:

- Dashboard client test calls `/api/runs/{id}/dashboard`.
- Inquiry mapper test extracts final values for Needs Met, Severe Failures, Trust, Inequality, and Administrative Load.
- Missing metric test renders unavailable state without crashing.
- Event timeline test sorts events by tick.
- Assumptions mapper test combines dashboard run facts with Charterfall session state.
- Boundary-copy test verifies assumption-bound language is present.

## Non-Goals

This slice should not implement:

- rerun with amended clauses
- before/after comparison
- final win/fail campaign rules
- citizen story cards
- faction testimony
- AI-generated inquiry summaries
- stress, sweep, robustness, or sensitivity views
- a single universal civic score

## Acceptance Checklist

This slice is complete when:

- Inquiry loads dashboard data for the current run ID.
- The five player-facing metrics are visible and traceable to raw PolityKit metric names.
- The event timeline or event empty state is visible.
- Severe-failure, collapse, or warning/watch signals appear when source data supports them.
- The assumptions block lists scenario, seed, model, parameters, chapter, and charter summary.
- The screen explains outcomes without requiring raw JSON.
- Inquiry copy uses assumption-bound language.
- Dashboard load failures are recoverable and do not corrupt run or charter state.
