# PolityKit v0.6 Implementation Plan

This plan is based on the current roadmap, README, CLI/API sweep implementation, and existing run summary contracts.

Version 0.6 should build on the existing v0.5-style sweep foundation rather than starting fresh. The current code already supports basic parameter sweeps, reruns, comparisons, and final-metric best/worst summaries. The v0.6 work should add analysis on top: batch stress sweeps, sensitivity scoring, collapse-threshold detection, robustness summaries, and challenge packs.

## 1. Consolidate Sweep Infrastructure

Move shared sweep logic out of both `PolityKit.Sim.Cli` and `PolityKit.Sim.Api` into a reusable analysis/service layer, likely a new `PolityKit.Sim.Analysis` project.

Deliverables:

- Shared parameter-combination builder.
- Shared sweep limits and validation.
- Shared final-metric extraction.
- Shared best/worst metric summarization.
- Tests proving CLI and API use the same sweep semantics.

## 2. Add Batch Stress Sweep Support

Extend sweeps from "one scenario plus parameters" to batch stress runs across scenarios, models, seeds, and parameter ranges.

Deliverables:

- `StressSweepRequest` and `StressSweepResult` domain types.
- CLI command, likely `stress`, separate from the existing `sweep` command.
- API endpoint, likely `POST /api/runs/stress`.
- Support for:
  - multiple scenarios,
  - multiple seeds,
  - multiple models,
  - named parameter grids,
  - hard run-count limits.
- Output bundle:
  - `stress-summary.json`
  - `stress-metrics.csv`
  - per-run bundles or persisted run IDs.

## 3. Define Failure And Recovery Criteria

Collapse detection needs explicit, configurable definitions so the system does not smuggle in assumptions.

Suggested defaults:

- `Needs Met < 0.5`
- `Trust < 25`
- `Severe Failures >= population-sensitive threshold`
- `Administrative Load >= configured threshold`
- Optional recovery: metric returns above or below threshold for N ticks.

Deliverables:

- `FailureCriterion` model.
- `CollapseEvent` or `ThresholdCrossing` result types.
- First-collapse tick per model, run, and metric.
- Recovery detected or "no recovery observed".
- Tests around boundary values and "never collapsed" cases.

## 4. Implement Sensitivity Reports

Use completed sweep results to rank which parameters most affect outcomes.

Initial methods should stay simple and inspectable:

- Per-parameter min/max outcome range.
- Delta from baseline/default.
- Direction of effect where monotonic.
- Basic correlation-like score for numeric parameter values.
- Separate results per model, scenario, and metric.

Deliverables:

- `SensitivityReport`
- `ParameterSensitivity`
- `MetricSensitivity`
- CLI JSON output.
- API response fields.
- Tests with synthetic sweep results where the expected ranking is obvious.

## 5. Add Model Robustness Summaries

Aggregate stress results into model-facing summaries.

Suggested fields:

- Scenarios tested.
- Seeds tested.
- Runs completed.
- Collapse rate.
- Median and earliest collapse tick.
- Recovery rate.
- Worst affected metric.
- Most sensitive parameter.
- Best and worst scenario names.

Deliverables:

- `ModelRobustnessSummary`
- Included in `stress-summary.json`.
- Included in API stress response.
- README examples showing how to interpret it carefully.

## 6. Add Scenario Challenge Packs

Create curated v0.6 examples under `examples/challenge-packs/` or `examples/stress/`.

Starter packs:

- Escalating food scarcity.
- Medicine shock severity ladder.
- Corruption plus administrative overload.
- Housing displacement with recovery window.
- Mixed cascading crisis.

Deliverables:

- JSON scenario files.
- Documentation in `examples/README.md` and `docs/politykit/scenarios.md`.
- Tests that every challenge-pack scenario validates against the schema.

## 7. Documentation And Acceptance Tests

Update the roadmap once v0.6 is implemented, and add focused tests rather than a large golden-output dependency.

Acceptance criteria:

- A user can run one command and receive a ranked sensitivity report.
- A user can see which parameter most affects collapse or recovery.
- A user can compare model robustness across challenge scenarios.
- API and CLI produce equivalent analysis shapes.
- Existing v0.5 sweep behavior remains compatible.

## Recommended Build Order

1. Consolidate sweep logic.
2. Add threshold detection.
3. Add sensitivity scoring.
4. Add robustness summaries.
5. Add challenge packs and docs.

This order gives each later piece real data to stand on while keeping the implementation incremental and testable.
