# PolityKit v0.7 Implementation Plan

This plan is based on the current roadmap, README, v0.6 stress/sensitivity implementation, model catalog, and existing simulation contracts.

Version 0.7 should move PolityKit from three single-mechanism allocation models toward composable governance bundles. The current code already supports deterministic runs, model parameters, manifests, sweeps, stress sweeps, collapse detection, sensitivity reports, and robustness summaries. The v0.7 work should add a richer model construction layer while preserving the existing mechanical baseline models for comparison.

## Pre-Work: Current Code Concerns

Before or during v0.7, address a few review findings in the existing v0.6-era surface. These are not blocking the current test suite, but they become more important once governance presets add more public inputs and more model implementations.

Deliverables:

- Harden sweep and stress run-count validation by computing Cartesian product sizes with `long` or `checked` arithmetic and failing as soon as the configured run limit is exceeded.
- Split CLI scenario path loading from API scenario resolution, or restrict API scenario paths to an allowed scenario directory so HTTP clients cannot request arbitrary server-local JSON paths.
- Validate API and CLI tick overrides after `WithTicks`, rejecting zero or negative ticks instead of silently producing empty runs.
- Clamp resource consumption in `DefaultWorldRule` to the citizen's remaining need so future composite models cannot over-consume resources when they emit larger allocation amounts.
- Add regression tests for overflow-sized sweep definitions, invalid tick overrides, API scenario path restrictions, and over-allocation behavior.

## 1. Define Governance Dimension Contracts

Add explicit domain types for the dimensions that make up a governance preset. These should be descriptive and composable rather than hard-coded as political labels.

Suggested dimensions:

- Allocation mechanism.
- Decision authority.
- Accountability mechanism.
- Information flow.
- Property regime.
- Appeal process.

Deliverables:

- `GovernanceProfile` domain type.
- Dimension value types or enums with stable IDs and display names.
- Optional per-dimension parameter bag for tunable weights.
- Validation that required dimensions are present.
- Tests for valid, invalid, and partially specified profiles.

## 2. Add A Composite Governance Model

Introduce a model that implements `ISystemModel` by combining governance dimensions into one `SystemDecision`.

The first version should stay inspectable: each dimension should contribute clear, limited effects to allocation priority, policy changes, or institutional actions. Avoid a large opaque scoring formula.

Deliverables:

- `CompositeGovernanceModel` or similarly named implementation.
- Dimension contribution helpers for allocation priority and institutional behavior.
- Deterministic behavior with the existing seeded `SystemContext`.
- Manifest support that describes the selected preset and dimension assumptions.
- Tests showing that changing one dimension changes behavior in an expected direction.

## 3. Build Governance Presets

Create named presets as reusable bundles of governance dimensions. Presets should be mechanism-based and documented as simplified experimental assumptions, not real-world claims.

Starter presets:

- Participatory commons.
- Regulated market.
- Central planning.
- Patronage hierarchy.
- Mutual aid federation.
- Technocratic administration.

Deliverables:

- `GovernancePresetCatalog`.
- Stable preset IDs, names, descriptions, assumptions, and known failure modes.
- Mapping from preset IDs to `GovernanceProfile`.
- Tests for catalog lookup by exact and kebab-case name.
- Tests that every preset can run against at least one existing scenario.

## 4. Integrate Presets With The Model Catalog

Expose composite presets through the existing model selection flow without breaking current baseline model names.

Possible approaches:

- Register one `CompositeGovernanceModel` per preset in `DefaultModelSet`.
- Add a catalog layer that resolves `preset:<id>` into a configured composite model.
- Keep baseline names such as `need-based-allocation` available for direct comparison.

Deliverables:

- CLI model selection support for governance presets.
- API model listing that includes preset metadata.
- API run requests that can select presets by stable ID.
- Backward-compatible behavior for the three baseline allocation models.
- Tests for CLI/API catalog resolution.

## 5. Extend Manifests For Composite Assumptions

Current model manifests describe a single model's assumptions. Composite presets need to explain both the overall preset and each dimension's contribution.

Deliverables:

- Manifest fields or conventions for governance dimensions.
- Per-dimension assumptions in API model responses.
- Known failure modes inherited from both preset-level and dimension-level definitions.
- Documentation that distinguishes baseline models from composite governance presets.
- Tests that preset manifests expose useful, non-empty assumptions.

## 6. Add Comparison Workflows For Baselines Versus Presets

Use existing run, sweep, stress, sensitivity, and robustness machinery to compare presets against the mechanical baseline models.

Deliverables:

- README examples comparing one preset against `NeedBasedAllocation`, `MarketBasedAllocation`, and `HierarchyBasedAllocation`.
- Stress example covering multiple presets and existing challenge scenarios.
- Output examples showing that `modelRobustness` works for presets.
- Acceptance tests that a mixed baseline-plus-preset stress run produces comparable summary shapes.

## 7. Document Interpretation Boundaries

Composite governance presets increase the risk that users read labels too literally. Documentation should keep the focus on mechanisms and assumptions.

Deliverables:

- Roadmap update once v0.7 is implemented.
- README section for governance presets.
- `docs/politykit/add-model.md` update explaining when to add a baseline model versus a preset dimension.
- Optional `docs/politykit/governance-presets.md` reference.
- Wording that clearly states presets are simplified bundles of assumptions, not claims about real societies.

## 8. Acceptance Tests

Add focused tests around composition rather than relying on large golden outputs.

Acceptance criteria:

- A user can list governance presets from the CLI/API.
- A user can run a preset by stable ID.
- A user can compare a preset against the existing baseline models.
- A user can stress-test multiple presets and receive robustness summaries.
- Preset manifests expose their dimensions, assumptions, and failure modes.
- Existing v0.6 sweep and stress behavior remains compatible.

## Recommended Build Order

1. Define `GovernanceProfile` and dimension contracts.
2. Implement preset catalog and validation.
3. Add the composite governance model.
4. Register presets in the model catalog.
5. Extend API/CLI model metadata.
6. Add mixed baseline-plus-preset stress tests.
7. Update README, roadmap, and model authoring docs.

This order keeps the new abstractions small and testable while preserving the existing v0.6 analysis pipeline as the proof that presets are comparable.
