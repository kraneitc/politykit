# PolityKit v0.8 Implementation Plan

This plan is based on the current roadmap, the v0.7 composite governance preset work, the existing CLI/API run inspection surface, stress/sensitivity/robustness outputs, and PolityKit's core reproducibility principle.

Version 0.8 should add optional AI-assisted exploration around already reproducible simulation outputs. AI should help users summarize, inspect, critique, and suggest next experiments, but it must not invisibly change scenarios, model behavior, metrics, seeds, or run results. The deterministic simulation stack remains the source of truth.

## Pre-Work: AI Boundary And Safety Decisions

Before adding AI features, define the boundaries that keep PolityKit honest and reproducible.

Deliverables:

- A documented rule that AI analysis is optional and never required to run simulations.
- A documented rule that AI output is advisory text or proposed artifacts, not authoritative simulation data.
- A stable way to record whether AI analysis was used, which input run IDs or files it read, and which provider/model generated the text.
- A privacy note for run data sent to external providers.
- Tests that existing CLI/API run, sweep, stress, and comparison workflows work without AI configuration.

## 1. Define AI Analysis Contracts

Create small contracts for AI-assisted analysis that sit above the existing run outputs.

Suggested concepts:

- `AiAnalysisRequest`.
- `AiAnalysisResult`.
- `AiAnalysisArtifact`.
- `AiAnalysisProvenance`.
- `AiAnalysisKind`.

Suggested analysis kinds:

- Run summary.
- Scenario suggestion.
- Model critique.
- Batch anomaly report.

Deliverables:

- Domain contracts in an appropriate project, likely `PolityKit.Sim.Analysis` or a new optional AI-focused project if dependencies require separation.
- Provenance fields for source run IDs, scenario names, model names, seeds, metric names, prompt template version, provider name, provider model, and creation time.
- Clear result fields for generated text, structured findings, warnings, and suggested follow-up commands.
- Tests for serialization-friendly result shapes and provenance completeness.

## 2. Build Deterministic Context Builders

AI prompts should be generated from deterministic, inspectable context builders. The prompt input should be reviewable before it is sent to a provider.

Deliverables:

- Context builder for a single run detail or CLI run bundle.
- Context builder for run comparison responses.
- Context builder for stress summaries, including sensitivity, collapse events, and `modelRobustness`.
- Context builder output that redacts or excludes fields that should not be sent externally.
- Tests that context builders include scenario, seed, model versions, selected parameters, final metrics, notable events, and relevant assumptions.
- Tests that context builders avoid unstable ordering.

## 3. Add Provider Abstraction And Local Disabled Mode

Introduce an AI provider abstraction without making any provider mandatory.

Deliverables:

- `IAiAnalysisProvider` or similarly named interface.
- A disabled provider that returns a clear "AI analysis is not configured" result.
- Configuration options for enabling AI analysis and selecting a provider.
- Timeouts, cancellation support, and bounded input/output sizes.
- Tests for disabled mode, provider failures, timeout handling, and deterministic error responses.

Implementation notes:

- Keep provider packages isolated so baseline simulation projects do not need AI-specific dependencies.
- Avoid logging prompt contents by default.
- If a cloud provider is added, keep API keys in configuration or environment variables, never in examples.

## 4. Add AI-Generated Run Summaries

Use the existing interpreted run data to generate concise narrative summaries.

The summary should explain what happened in the run, which metrics moved, which shocks or events may be relevant, and which assumptions were active. It should not claim real-world predictive validity.

Deliverables:

- CLI command or option for generating an AI summary from a completed run bundle.
- API endpoint for generating an AI summary for a stored run.
- Stored or exportable AI summary artifact with provenance.
- Tests using a fake provider to verify request context and response mapping.
- Documentation examples showing summary output as advisory interpretation.

## 5. Add AI-Assisted Scenario Suggestions

Use run outcomes and existing scenario schema constraints to propose new scenario variants.

The output should be a suggested scenario draft or parameterized change set, not an automatically accepted scenario file.

Deliverables:

- Scenario-suggestion analysis kind.
- Context builder that includes the source scenario, validation rules, stress results, and observed failures.
- Suggested scenario payload shape that can be reviewed before saving.
- Validation step that runs suggested scenarios through existing scenario validation before they can be written.
- CLI/API workflows that clearly label generated scenarios as drafts.
- Tests that invalid provider output is rejected or reported without writing files.

## 6. Add AI-Assisted Model Critique

Use manifests, governance dimensions, metric outcomes, collapse events, and robustness summaries to critique model assumptions.

The critique should identify possible blind spots, missing assumptions, suspicious behavior, or follow-up experiments. It should not rewrite model code.

Deliverables:

- Model-critique analysis kind.
- Context builder for one or more model manifests plus relevant run/stress results.
- Critique result fields for assumption risks, observed failure modes, suggested tests, and suggested documentation updates.
- Tests with fake provider output covering baseline models and composite governance presets.
- Documentation that critiques are prompts for human review, not proof that a model is correct or incorrect.

## 7. Add Batch Anomaly Detection Assistance

Use stress, sweep, and comparison outputs to identify surprising or unstable patterns across many runs.

AI should operate on existing deterministic summaries such as final metrics, sensitivity reports, collapse events, and `modelRobustness`, rather than raw hidden state.

Deliverables:

- Batch anomaly analysis kind.
- Context builder for `stress-summary.json` and API stress responses.
- Structured anomaly output fields for model, scenario, seed, metric, observed value, and explanation.
- Tests showing that anomaly requests include mixed baseline and governance preset runs.
- Tests that provider output cannot invent run IDs or models that were not in the source context without being flagged.

## 8. Integrate With API Without Breaking Existing Endpoints

Expose AI analysis through additive endpoints that do not change existing run, sweep, stress, or model response contracts.

Possible endpoints:

- `POST /api/runs/{id}/ai/summary`
- `POST /api/runs/{id}/ai/scenario-suggestions`
- `POST /api/runs/{id}/ai/model-critique`
- `POST /api/runs/stress/ai/anomalies`

Deliverables:

- API contracts for AI analysis requests and responses.
- ProblemDetails-compatible failures for disabled provider, missing runs, invalid provider output, or oversized context.
- API tests using a fake provider.
- No behavior changes to existing run creation, run lookup, comparison, sweep, or stress endpoints.

## 9. Integrate With CLI And File Outputs

Add local workflows for users who primarily work with run bundles.

Possible commands:

- `ai-summary --run <directory>`
- `ai-suggest-scenario --run <directory> --out <file>`
- `ai-critique-model --run <directory> --model <name>`
- `ai-anomalies --stress-summary <file>`

Deliverables:

- CLI commands or options for each accepted v0.8 workflow.
- Output files such as `ai-summary.json`, `ai-scenario-suggestions.json`, `ai-model-critique.json`, and `ai-anomalies.json`.
- Provenance embedded in each output file.
- Tests for CLI argument validation and output shape where practical.
- README examples for local use.

## 10. Document Interpretation Boundaries

AI features increase the risk that users over-trust generated text. Documentation should be direct and repetitive about limits.

Deliverables:

- README section for optional AI-assisted exploration.
- API examples showing disabled mode and configured mode.
- CLI examples using completed run bundles.
- Documentation that AI outputs are generated interpretations of recorded simulation artifacts.
- Documentation that AI outputs do not alter run results, model behavior, metric calculations, or scenario validation.
- Known limitations text for hallucinations, provider drift, privacy, prompt sensitivity, and non-authoritative interpretation.

## 11. Acceptance Tests

Add focused tests around optionality, provenance, and non-interference.

Acceptance criteria:

- A user can run all existing simulations without configuring AI.
- A user can generate an AI summary for a stored run using a fake provider.
- A user can generate a scenario suggestion draft and validate it before saving.
- A user can request model critique for both baseline models and governance presets.
- A user can request anomaly assistance for a stress result that includes mixed baseline-plus-preset models.
- AI analysis artifacts include source run/scenario/model/seed provenance and provider metadata.
- AI-disabled mode returns a clear, non-crashing response.
- Existing deterministic run, sweep, stress, comparison, and model catalog tests remain compatible.

## Recommended Build Order

1. Define AI analysis contracts and provenance.
2. Build deterministic context builders for run, comparison, and stress outputs.
3. Add provider abstraction plus disabled and fake providers.
4. Add API run-summary workflow with fake-provider tests.
5. Add CLI run-summary workflow and output artifact.
6. Add scenario suggestion workflow with validation gate.
7. Add model critique workflow for baselines and presets.
8. Add batch anomaly workflow for stress summaries.
9. Update README, API examples, CLI examples, and limitation docs.
10. Run the full build/test gate.

This order keeps AI outside the simulation core, proves non-interference early, and lets later workflows reuse the same contracts, context builders, provider abstraction, and provenance rules.
