# PolityKit v1.0 Implementation Plan

This plan is based on the current roadmap, README, v0.8 optional AI-assisted exploration work, API/CLI run inspection surfaces, contributor docs, and the current release-readiness review.

Version 1.0 should turn the existing working simulation stack into a stable, documented, extensible public framework release. The current repo already has the core engine, model and metric contracts, reproducible run bundles, API-backed inspection, examples, governance presets, stress analysis, optional AI-assisted review, and contributor documentation. The v1.0 work should harden those surfaces into an explicit public contract rather than adding a large new feature area.

## Pre-Work: Release Boundary Decisions

Before changing code, decide what "stable" means for v1.0.

Deliverables:

- A written compatibility policy for public engine, model, metric, scenario, CLI output, and API response contracts.
- A list of public contracts that are considered stable in v1.0.
- A list of experimental or advisory surfaces that remain outside the v1.0 compatibility promise.
- A release checklist covering build, test, docs, examples, packaging, and smoke verification.
- A decision on whether v1.0 is a source/framework release only, or also a packaged NuGet/CLI release.

Suggested v1.0 stable surfaces:

- `PolityKit.Sim.Core` model, metric, scenario, world, event, and simulation request/result contracts.
- `PolityKit.Sim.Engine` deterministic run behavior for seeded scenarios.
- CLI run bundle files: `config.json`, `summary.json`, `metrics.csv`, `events.jsonl`, `citizens-final.csv`, and `ai-analysis.json`.
- API endpoints for model/metric/scenario discovery, run creation, run lookup, metrics/events/dashboard inspection, rerun, compare, sweep, and stress.
- Contributor workflows for adding models, metrics, and scenarios.

Suggested experimental or advisory surfaces:

- Optional AI-assisted outputs and provider integrations.
- Governance preset labels as simplified experimental bundles, not real-world claims.
- Any future frontend dashboard or visualization layer.

## 1. Stabilize Public Core Contracts

Audit the public framework contracts that external contributors will use.

Deliverables:

- Review `ISystemModel`, `IMetric`, `ScenarioDefinition`, `SimulationRunRequest`, `SimulationRunResult`, `ModelRunResult`, events, world state, and random-source contracts.
- Document which contracts are safe for external model and metric authors to depend on.
- Add XML documentation or focused markdown notes for non-obvious extension points.
- Add tests that protect deterministic behavior for custom model and custom metric implementations.
- Ensure nullability is enabled and public contracts fail predictably on invalid input.
- Decide whether model versions should move from `0.1.0` to `1.0.0`, and update tests/docs if so.

Acceptance checks:

- A small test-only external model can run through the engine without internal project hooks.
- A small test-only metric can run through the engine and appear in CLI/API outputs.
- Seeded runs remain deterministic for the same scenario, model set, metric set, parameters, and version.

## 2. Lock Down Run Bundle And JSON Contracts

Treat run bundles as public artifacts that users can inspect, archive, and compare.

Deliverables:

- Document the v1.0 run bundle file set and the purpose of each file.
- Add or update JSON examples for `config.json`, `summary.json`, `sweep-summary.json`, `stress-summary.json`, and AI sidecars.
- Add schema or contract tests for the most important JSON output shapes.
- Confirm `summary.json` includes scenario, seed, ticks, model names, model versions, final metrics, event counts, and notable metric changes.
- Confirm sweep and stress summaries include run directories, parameter combinations, final metrics, best/worst summaries, sensitivity, collapse events, and `modelRobustness` where applicable.
- Clearly mark AI artifacts as advisory and separate from deterministic simulation data.

Acceptance checks:

- A generated run bundle can be inspected without rebuilding the project.
- A generated stress summary can be used by the AI anomaly workflow without rerunning stress.
- Golden/example bundles match the documented v1.0 shape.

## 3. Harden CLI Workflows

Make the CLI reliable enough for public examples and contributor workflows.

Deliverables:

- Review `run`, `sweep`, `stress`, `list-models`, AI advisory commands, and help text for consistency.
- Ensure CLI examples are PowerShell-friendly and Unix-shell friendly where practical.
- Add smoke tests for documented commands that generate run, sweep, stress, and AI advisory artifacts.
- Ensure invalid arguments return clear non-zero failures without partial or misleading output.
- Decide whether the CLI should be packaged as a .NET tool for v1.0; if yes, add package metadata and a packaging smoke test.

Acceptance checks:

- README CLI examples run as written.
- Invalid scenario, model, tick, sweep, stress, and AI inputs fail clearly.
- Generated output paths are predictable and documented.

## 4. Harden API Inspection Workflows

The roadmap allows either a dashboard or API-backed inspection workflow. For v1.0, the API should be treated as the primary inspection surface.

Deliverables:

- Confirm all documented API endpoints are covered by integration tests.
- Review OpenAPI output and Swagger UI compatibility.
- Document disabled and configured AI API behavior as advisory and optional.
- Ensure API failures use `ProblemDetails` consistently for invalid run, rerun, sweep, stress, and AI requests.
- Confirm file-backed run storage is documented, configurable, and safe for normal local use.
- Decide whether API route and response contracts are stable for v1.0 or explicitly versioned as unversioned local API.

Acceptance checks:

- A user can create a run, inspect it, rerun it, compare it, sweep it, and stress it through documented API calls.
- Existing run, sweep, stress, comparison, and model catalog tests pass without AI configuration.
- OpenAPI is available in development mode and remains Swagger UI compatible.

## 5. Finalize Contributor Documentation

External contributors should be able to add models, metrics, and scenarios without reading the whole codebase first.

Deliverables:

- Review and update:
  - `docs/politykit/contributing.md`.
  - `docs/politykit/add-model.md`.
  - `docs/politykit/add-metric.md`.
  - `docs/politykit/add-scenario.md`.
  - `docs/politykit/scenarios.md`.
  - `docs/politykit/governance-presets.md`.
  - `docs/politykit/ai-boundaries.md`.
- Add a short "v1.0 extension points" section linking to model, metric, and scenario docs.
- Ensure docs name the relevant project, interface, catalog, tests, and example command for each contribution type.
- Ensure guidance explains assumptions, limitations, and interpretation boundaries.
- Add markdown link checks or at least a lightweight link audit for local documentation links.

Acceptance checks:

- A new contributor can add a model by following one page.
- A new contributor can add a metric by following one page.
- A new contributor can add a scenario by following one page and validating it against the schema.

## 6. Release Metadata, CI, And Packaging

Public v1.0 needs repeatable verification outside a local Codex session.

Deliverables:

- Add CI workflow files under `.github/workflows` for restore, build, and test.
- Add shared project metadata where appropriate:
  - Version.
  - Authors.
  - Description.
  - Repository URL.
  - License expression or license file.
  - Package tags.
- Decide which projects are packable for v1.0.
- Add package smoke checks if NuGet packages or a .NET tool are part of the release.
- Add a `CHANGELOG.md` or release notes file summarizing the v1.0 public contract and known limitations.
- Document supported SDK/runtime version.

Acceptance checks:

- CI runs `dotnet build PolityKit.slnx` and `dotnet test PolityKit.slnx --no-restore` successfully.
- Packable projects produce packages with correct metadata, if packaging is in scope.
- Release notes clearly state what is stable, what is experimental, and what is not intended for real-world policy authority.

## 7. Review Examples And Golden Artifacts

Examples are part of the public framework story.

Deliverables:

- Confirm every JSON file under `examples/` is valid and loadable.
- Confirm example stress grids and scenario files still run with current model catalogs.
- Update `examples/README.md` to match current CLI commands and output files.
- Refresh the golden interpreted run bundle if output shapes changed.
- Add tests that protect example scenario compatibility and golden bundle shape.

Acceptance checks:

- Every documented example command runs successfully.
- Example outputs demonstrate baseline models, governance presets, sweep, stress, and run interpretation.
- Golden artifacts are current, intentional, and easy to inspect.

## 8. Confirm Limitation And Safety Messaging

The project must clearly communicate its purpose and limits before public release.

Deliverables:

- Review README framing: PolityKit is an assumption laboratory, not policy authority.
- Review model and governance preset wording for neutrality and non-real-world claims.
- Review AI boundary language for hallucinations, provider drift, privacy, prompt sensitivity, and non-authoritative interpretation.
- Ensure generated AI examples and fake-provider output do not imply predictive validity.
- Add a short release-note warning that v1.0 is a simulation framework release, not empirical validation.

Acceptance checks:

- README, governance preset docs, and AI boundary docs all repeat the non-authoritative interpretation rule.
- No public example claims a model, preset, or AI output proves a real-world system is better.

## 9. Final v1.0 Verification Gate

Run the complete release gate from a clean checkout or clean working tree.

Required checks:

- `dotnet restore PolityKit.slnx`.
- `dotnet build PolityKit.slnx`.
- `dotnet test PolityKit.slnx --no-restore`.
- CLI smoke: run, sweep, stress, AI summary with fake provider, scenario suggestion with fake provider, model critique with fake provider, anomaly report from stress summary.
- API smoke: create run, inspect run, rerun, compare, sweep, stress, disabled AI response, fake-provider AI summary.
- Documentation audit for local links and stale "not built yet" statements.
- Package or release artifact smoke checks if packaging is in scope.

Success criteria:

- External contributors can add models, metrics, and scenarios using documented extension points.
- Runs are reproducible and inspectable through local files and the API.
- The public contract is explicit enough that future breaking changes can be identified.
- The project clearly communicates its limitations, assumptions, and non-authoritative purpose.
- CI and local verification agree.

## Recommended Build Order

1. Write the compatibility policy and release checklist.
2. Audit public core/model/metric/scenario contracts.
3. Lock down run bundle and API response contracts with focused tests.
4. Harden CLI and API examples.
5. Update contributor docs and example artifacts.
6. Add CI and release/package metadata.
7. Add release notes or changelog.
8. Run the full v1.0 verification gate.
